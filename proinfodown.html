视频下载工具 — 需求说明书

版本: v1.0
日期: 2026-07-17
状态: 待评审

1. 背景与目标

1.1 背景

抖音和 X（Twitter）是目前最常用的短视频/社交视频平台，用户经常有保存视频到本地的需求。但两个平台存在以下限制：

  • 抖音：App 内无直接下载无水印视频的入口，网页端需要登录才能查看内容
  • X（Twitter）：部分媒体内容需要登录才能访问，且没有官方下载入口

市面上的下载工具多为在线网站（不稳定、有广告）或桌面软件（不可远程访问），缺少一个可自部署、Web 访问、支持登录态管理的轻量方案。

1.2 目标

构建一个 可 Docker 部署的 Web 视频下载工具，支持：

  1. 通过浏览器粘贴视频链接，下载抖音和 X 的视频
  2. 支持配置平台登录 Cookie，解决需要登录才能访问的问题
  3. 下载结果可选择保存到本地浏览器下载或服务端指定路径
  4. 提供下载历史记录和文件管理

1.3 非目标（本期不做）

  • 不做批量下载（一次只处理一个链接）
  • 不做视频转码/格式转换
  • 不做用户认证/多用户支持（内网工具，单用户）
  • 不做视频编辑/裁剪
  • 不支持其他平台（B站、YouTube 等，后续可扩展）

2. 用户角色

角色 描述
使用者 通过浏览器访问工具，粘贴链接下载视频的单一用户（工具部署者本人）

5. 用户场景

场景 1：首次使用配置 Cookie

用户部署 Docker → 浏览器访问 → 进入设置页面 → 
粘贴抖音 Cookie → 保存 → 粘贴 X Cookie → 保存 → 开始使用

场景 2：下载抖音视频到本地

用户复制抖音视频链接 → 粘贴到下载框 → 自动识别平台 →
系统预解析显示视频信息 → 选择「下载到本地」→ 点击下载 →
实时显示进度 → 完成后浏览器自动下载视频文件

场景 3：下载 X 视频到服务端

用户复制 X 视频链接 → 粘贴到下载框 → 自动识别平台 →
系统预解析显示视频信息 → 选择「保存到服务端」→ 点击下载 →
实时显示进度 → 完成后视频保存在服务端配置路径 → 历史记录中出现该条目

场景 4：查看历史并重新下载

用户进入历史页面 → 查看下载记录列表 → 
点击某条记录的「重新下载」→ 系统使用原链接发起新下载

场景 5：Cookie 失效后更新

用户下载失败 → 错误提示「Cookie 可能已过期」→
进入设置页面 → 更新 Cookie → 重新下载

6. 约束条件

  1. 核心下载引擎：使用 yt-dlp，它已内置抖音和 X 的提取器，支持 Cookie 认证
  2. 运行环境：Docker 容器内需安装 ffmpeg（用于视频合并）
  3. 网络要求：服务端需能访问抖音和 X 的服务器
  4. Cookie 获取：用户需自行从浏览器获取 Cookie（工具不提供自动登录）
  5. 文件格式：两个平台的视频原始格式均为 MP4/H.264/AAC，无需额外转码；高画质版本可能音视频流分离，通过 ffmpeg 合并后统一输出 MP4

7. 术语表

术语 说明
Cookie 浏览器登录凭证，用于让下载工具模拟已登录状态
Netscape 格式 一种标准的 Cookie 文件格式，yt-dlp 原生支持
yt-dlp 开源视频下载命令行工具，支持 1000+ 网站
本地下载 视频下载到服务端内存/临时文件后，通过 HTTP 传输给浏览器保存
服务端下载 视频直接下载到服务端的指定目录持久保存

1.3 视频格式说明

抖音和 X 的视频原始格式情况：

平台 容器格式 视频编码 音频编码 分发方式
抖音 MP4 H.264 (avc1) AAC 多为单文件，高画质可能分离音视频流
X (Twitter) MP4 H.264 (avc1) AAC 多为单文件，高画质可能分离音视频流

结论：两个平台本身就是 MP4/H.264 格式，无需额外格式转换。

但需处理音视频流分离的情况（高画质版本常见），yt-dlp 配置策略：

ydl_opts = {
    # 优先选择 H.264 编码 + 最佳音频，兼容性最好
    'format': 'bv*[vcodec=avc1]+ba/b',
    # 合并后统一输出 MP4 容器
    'merge_output_format': 'mp4',
}
  • bv*[vcodec=avc1]+ba：优先选 H.264 视频流 + 最佳音频流，回退到最佳单文件
  • merge_output_format: mp4：分离流合并后强制输出 MP4
  • ffmpeg 负责合并操作，无需额外转码

1.4 请求流转

用户浏览器
   │
   │ 1. 粘贴视频链接 → POST /api/parse
   │    FastAPI 调用 yt-dlp extract_info (extract=False)
   │    返回视频信息 (标题/作者/缩略图/时长)
   │
   │ 2. 点击下载 → POST /api/download
   │    FastAPI 创建下载任务，返回 task_id
   │    后台线程执行 yt-dlp.download()
   │
   │ 3. 进度获取 → GET /api/download/{task_id}/progress (SSE)
   │    Server-Sent Events 推送实时进度
   │
   │ 4a. 本地下载 → GET /api/download/{task_id}/file
   │     返回文件流，浏览器触发下载
   │
   │ 4b. 服务端下载 → 文件已保存到服务端路径
   │     记录写入 history.json
   │
   ▼

// Request
{
  "url": "https://www.douyin.com/video/7xxx"
}

// Response 200
{
  "platform": "douyin",
  "title": "视频标题",
  "uploader": "作者",
  "thumbnail": "https://...",
  "duration": 30

{

// Request
{
  "url": "https://www.douyin.com/video/7xxx",

{
  "task_id": "a1b2c3d4e5f6"

{
  "error": "抖音 Cookie 未配置，请先在设置中配置"

// Server-Sent Events 流
data: {"status":"downloading","progress":45.2,"speed":"3.5 MiB/s","eta":"12s"}

data: {"status":"downloading","progress":78.1,"speed":"4.1 MiB/s","eta":"5s"}

data: {"status":"completed","progress":100,"filename":"搞笑猫咪.mp4","filesize":8912896}

// 或错误
data: {"status":"error","error":"Cookie 已过期"}

    url: str              # 视频链接

    target: str           # "local" | "server"

3. 改进建议

3.1 格式选择策略优化（建议）

现状：概要设计使用 bv*[vcodec=avc1]+ba/b，在 X 平台上会触发 /b 回退到单文件格式。

原因：bv* 选择器会考虑包含音频的视频流，[vcodec=avc1] 过滤器排除了 vcodec 未知的 http-* 格式，导致 bv*[vcodec=avc1]+ba 匹配不到可用组合，回退到 /b。

建议：改为 bestvideo*[vcodec=avc1]+bestaudio/b 或 bestvideo+bestaudio/b：

ydl_opts = {
    # 方案A（推荐）：不限制编码，让 yt-dlp 自动选择最佳分离流
    'format': 'bestvideo+bestaudio/b',
    # 方案B：限制 H.264 但用 bestvideo 替代 bv*
    # 'format': 'bestvideo*[vcodec=avc1]+bestaudio/b',
    'merge_output_format': 'mp4',
}

验证依据：bestvideo+bestaudio/b 已在实际测试中成功选中 hls-927（avc1, 720p）+ hls-audio-128000 并用 ffmpeg 合并为 MP4。

3.2 时区配置（建议）

在 docker-compose.yml 中添加时区环境变量：

environment:
  - DATA_DIR=/data
  - TZ=Asia/Shanghai

3.3 Cookie 获取方式说明

现状：X (Twitter) 平台要求登录 Cookie 才能解析视频。Chrome 127+ 的 App-Bound Encryption 导致无法通过 browser_cookie3 或 yt-dlp 的 --cookies-from-browser 自动提取 Cookie。

建议：在 Web UI 中提供 Cookie 输入框，引导用户手动粘贴 Cookie 字符串。关键 Cookie 字段：
- auth_token — 登录认证令牌
- ct0 — CSRF 防护令牌

获取方法：浏览器登录 x.com → F12 开发者工具 → Network → 刷新页面 → 任意请求的 Request Headers → 复制 Cookie 值。

系统通过概要设计 5.2 节的 Cookie 转换逻辑，将字符串转为 Netscape 格式文件供 yt-dlp 使用。

3.4 yt-dlp 版本管理（建议）

考虑在 Dockerfile 中添加 yt-dlp 定期更新机制，或在 requirements.txt 中使用 >= 版本约束：
